3Ô∏è‚É£ Permission Structure (Very Important)

-rwx r-x r--
  ‚îÇ    ‚îÇ   ‚îÇ
  ‚îÇ    ‚îÇ   ‚îî‚îÄ‚îÄ Others
  ‚îÇ    ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ Group
  ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ User (Owner)

Permission letters:

  • w ‚Üí write
  • x ‚Üí execute
  • - ‚Üí no permission

4Ô∏è‚É£ User, Group, Others

TypeMeaning
UserFile owner
GroupGroup owner
OthersEveryone else

üëâ Linux always checks permissions in this order: User ‚Üí Group ‚Üí Others

5Ô∏è‚É£ File vs Directory Permissions

File

  • r ‚Üí read file
  • w ‚Üí modify file
  • x ‚Üí run file (script)

Directory

  • r ‚Üí list files
  • w ‚Üí create/delete files
  • x ‚Üí enter directory

üëâ Without x on a directory, you cannot cd into it.

6Ô∏è‚É£ Changing Permissions chmod (change mode)

Symbolic mode

chmod u+x file.sh              # add execute to user
chmod g-w file.txt             # remove write from group
chmod o+r file.txt             # add read to others
chmod u=rwx,g=rx,o=r file.txt  # set exact permissions
chmod a+x script.sh            # add execute to all
chmod a-r file.txt             # remove read from all
chmod a=r file.txt             # set read for all
chmod +x script.sh             # add execute to all
  • a means all (user, group, others)
  • u means user (owner)
  • g means group
  • o means others

Numeric mode (VERY IMPORTANT)

NumberPermission
4read
2write
1execute

Example:

Meaning:

User   ‚Üí 7 (rwx)
Group  ‚Üí 5 (r-x)
Others ‚Üí 5 (r-x)

üîü Common AWS Permission Issues

Linux User Management

üëà

üîë Big Picture: EC2 Access + Linux Users (ONE CLEAR MODEL)

When you create an EC2 instance, there are two separate systems involved:

1Ô∏è‚É£ AWS level (IAM & Key Pair)

Controls:

  • Who can connect to the EC2
  • Uses SSH keys (.pem)

2Ô∏è‚É£ Linux level (Users & Groups)

Controls:

  • What a person can do inside the server
  • Uses users, groups, permissions

üëâ These two are independent but work together.

üîê What the .pem file really does

  • A .pem file is NOT a user
  • It is NOT root access
  • It only proves:

‚ÄúI am allowed to connect to this server‚Äù

When you SSH using a PEM file:

ssh -i key.pem ec2-user@IP

You log in as:

ec2-user

‚úî Not root ‚úî Normal Linux user ‚úî Has sudo access

üë§ Are you root after login?

‚ùå No

AWS disables direct root login for security.

To become root:

sudo su -

or

sudo -i

Root access is temporary and controlled.

üë• Giving EC2 access to another person (CORRECT WAY)

You do NOT share your PEM file.

Correct process:

  1. Other person generates their own SSH key
  2. They send you only the public key
  3. You create a Linux user on EC2
  4. You add their public key to that user
  5. You set permissions for what they can access

üîë 1Ô∏è‚É£ What does ‚Äúlogin with a PEM file‚Äù actually mean?

When you run this command:

ssh -i mykey.pem ec2-user@EC2-IP

You are doing SSH key-based authentication.

Let‚Äôs break it down:

PartMeaning
sshSecure Shell (connect to server)
-i mykey.pemUse this private key to prove identity
ec2-user@EC2-IPLogin as Linux user ec2-user

üëâ Important: The PEM file is not a user. It is not root access. It is just a key to open the door.

üë§ 2Ô∏è‚É£ Which user do you become after login?

You become the Linux user you specify in the SSH command.

Example:

ssh -i mykey.pem ec2-user@EC2-IP

You are logged in as:

ec2-user

You can verify:

whoami

Output:

ec2-user

‚úÖ You are NOT root ‚úÖ You are a normal Linux user

‚ùå 3Ô∏è‚É£ Are you root by default after login?

NO. Never.

Even though you used a PEM file:

  • You are not root
  • You are not superuser
  • You have limited permissions

AWS does this intentionally for security.

üß† 4Ô∏è‚É£ Why AWS does NOT log you in as root

AWS disables direct root login by default.

Reasons:

  1. Security

    • Root can delete OS, disks, users

  2. Accidental damage

    • One wrong command can destroy server

  3. Industry best practice

    • Least privilege principle

So AWS says:

‚ÄúLogin as a normal user, become root only when needed.‚Äù

üîê 5Ô∏è‚É£ Then how can ec2-user do admin work?

Because ec2-user has sudo access.

Check:

groups ec2-user

Output:

ec2-user wheel

The wheel group allows:

sudo

üöÄ 6Ô∏è‚É£ How to access root (IMPORTANT)

There are two correct and safe ways.

‚úÖ Method 1: Run a single command as root

sudo command

Example:

sudo yum install nginx

What happens:

  • Only this command runs as root
  • You remain ec2-user

Check:

sudo whoami

Output:

root

‚úÖ Method 2: Switch to full root shell

sudo su -

OR

sudo -i

Now your prompt changes:

[root@ip-172-31-x-x ~]#

Now:

  • You are root
  • You have full control

To exit root:

exit

‚ö†Ô∏è 7Ô∏è‚É£ Why root login is dangerous

As root you can:

  • Delete /
  • Stop networking
  • Remove users
  • Break boot process

That‚Äôs why:

  • Root login via SSH is disabled
  • Root is accessed temporarily via sudo

üß† 8Ô∏è‚É£ Mental model (VERY IMPORTANT)

Think like this:

  • PEM file ‚Üí proves who you are
  • Linux user ‚Üí decides what you can do
  • sudo ‚Üí temporary root power
PEM file ‚â† root

‚úÖ Final clear statement (remember this)

When you log in using a PEM file, you are logged in as a normal Linux user (like ec2-user), not as root. AWS disables direct root login for security reasons. To perform administrative tasks, you temporarily become root using sudo.

How to create a new user and how to do proper setup for that user

This is exactly how it is done on real EC2 servers.

üß† First understand the goal

When you create a new user, you want:

‚úî The user to log in with their own SSH key ‚úî The user to have their own home directory ‚úî The user to access only allowed files ‚úî The user to NOT be root by default

üë§ Step 1: Login as admin user

First, you (admin) log in to EC2:

ssh -i admin.pem ec2-user@EC2-IP

You are logged in as:

ec2-user

This user has sudo access.

üë§ Step 2: Create a new Linux user

Create a new user (example: devuser):

sudo useradd devuser

What this does:

  • Creates a Linux user named devuser
  • Assigns a UID (user ID)
  • Creates entry in /etc/passwd

üìÇ Step 3: Create home directory (IMPORTANT)

Some systems create it automatically, but to be safe:

sudo mkdir /home/devuser

Set ownership:

sudo chown devuser:devuser /home/devuser

Now:

/home/devuser

belongs only to devuser.

üîê Step 4: Create .ssh directory

This is required for SSH key login.

sudo mkdir /home/devuser/.ssh

Set correct permissions:

sudo chmod 700 /home/devuser/.ssh
sudo chown devuser:devuser /home/devuser/.ssh

Why 700?

  • Only the user can read/write/enter
  • SSH will refuse login if permissions are wrong

üîë Step 5: Add public key (MOST IMPORTANT)

Create the authorized keys file:

sudo nano /home/devuser/.ssh/authorized_keys

Paste the public key of the user here.

‚ö†Ô∏è Only public key, never private key.

Set permissions:

sudo chmod 600 /home/devuser/.ssh/authorized_keys
sudo chown devuser:devuser /home/devuser/.ssh/authorized_keys

üß™ Step 6: Test user login

Now the user can log in from their own machine:

ssh -i devuser.pem devuser@EC2-IP

Check:

whoami

Output:

devuser

‚úÖ User is logged in ‚úÖ Not root ‚úÖ Isolated from other users

üîê Step 7: Decide sudo access (VERY IMPORTANT)

‚ùì Should the user be root-capable?

‚ùå If NO (most secure):

Do nothing.

User:

  • Cannot install software
  • Cannot change system config
  • Cannot access /etc, /var, etc.

‚úÖ If YES (admin / senior dev):

Add user to sudo group.

On Amazon Linux:

sudo usermod -aG wheel devuser

On Ubuntu:

sudo usermod -aG sudo devuser

Now user can run:

sudo command

üìÇ Step 8: Folder & file access control

Example: You have a folder:

/app-data

You want:

  • devuser ‚Üí access
  • Others ‚Üí no access
sudo chown devuser:devuser /app-data
sudo chmod 700 /app-data

Now:

  • Only devuser can access
  • Others are blocked

üë• Step 9: Groups (Professional setup)

Create a group:

sudo groupadd devteam

Add user to group:

sudo usermod -aG devteam devuser

Assign folder to group:

sudo chown :devteam /shared-data
sudo chmod 770 /shared-data

Now:

  • All devteam members can access
  • Others cannot

‚ùå Common mistakes (IMPORTANT)

‚ùå Forgetting .ssh permissions ‚ùå Putting private key on server ‚ùå Logging everyone in as ec2-user ‚ùå Giving sudo to everyone

üß† Mental model (REMEMBER THIS)

User = identity
Group = role
SSH key = login method
Permissions = access control

‚úÖ Final summary (Task 2)

To set up a new user on an EC2 instance, you create a Linux user, create a home directory, configure SSH access using the user‚Äôs public key, set strict permissions on .ssh files, and optionally grant sudo access using groups. This ensures secure, controlled access for each individual user.

How the other user creates their PEM file and shares it with you

This task is about what the OTHER PERSON does, not you.

üß† First, clear one very important misunderstanding

üëâ The admin does NOT create the PEM file for the user. üëâ The user creates their OWN PEM file on their OWN machine.

This is how it works in real companies.

üë§ Who is ‚Äúthe other user‚Äù?

  • Developer
  • Teammate
  • Intern
  • Ops engineer

They want access to your EC2 server.

üîë Step 1: User creates SSH key pair on their own machine

On their laptop / system, they run:

ssh-keygen

They will see something like:

Generating public/private rsa key pair.
Enter file in which to save the key (/home/user/.ssh/id_rsa):

They can press Enter OR give a custom name:

devuser.pem

What files are created?

Two files are created:

FilePurpose
devuser.pemPRIVATE KEY (never shared)
devuser.pem.pubPUBLIC KEY (shared with admin)

üìå The .pem file stays ONLY with the user.

üîê Step 2: Optional ‚Äì set passphrase (recommended)

User may be asked:

Enter passphrase (empty for no passphrase):
  • Passphrase = extra password protection
  • If someone steals the key ‚Üí still cannot use it

In companies: ‚úîÔ∏è Passphrase is recommended

üì§ Step 3: User shares ONLY the public key

The user sends ONLY this file (or its content):

devuser.pem.pub

Example content:

ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQC7...

They can share it via:

  • Email
  • Slack
  • Ticket
  • Git (private repo)

‚ö†Ô∏è They NEVER share:

devuser.pem

üîí Why private key is never shared

If someone has the private key:

  • They can login as that user
  • No password needed
  • Full access

So:

Private key = identity

üñ•Ô∏è Step 4: You (admin) add public key to EC2

You paste the public key into:

/home/devuser/.ssh/authorized_keys

Now the server trusts:

  • Anyone who proves they own the matching private key

üß™ Step 5: How the user logs in

From their machine, the user runs:

ssh -i devuser.pem devuser@EC2-IP

Important points:

  • Command runs on their laptop
  • Uses their private key
  • Logs in as devuser

üîç Step 6: How SSH verifies identity (Simple explanation)

  1. Server sends a challenge
  2. User signs it with private key
  3. Server checks public key
  4. Match found ‚úîÔ∏è
  5. Login allowed üéâ

The private key is never sent to the server.

‚ùå Common beginner mistakes (VERY IMPORTANT)

‚ùå Admin generates key and shares PEM ‚ùå User sends private key via WhatsApp üò¨ ‚ùå Storing private keys on server ‚ùå One key used by multiple users

üß† Simple analogy (remember this)

Real worldSSH
House keyPrivate key (.pem)
Lock designPublic key (.pub)
House ownerEC2 admin

‚úÖ Final summary (Task 3)

The other user generates their own SSH key pair on their own system using ssh-keygen. They keep the private key (.pem) securely with them and share only the public key (.pub) with the EC2 admin. The admin adds this public key to the user‚Äôs authorized_keys file, allowing secure SSH access without sharing secrets.
